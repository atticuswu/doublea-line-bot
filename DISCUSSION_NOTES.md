## 議題一：多群組邀請名單設計

### 情境
- 群組 A：Atticus + Angel（家庭）
- 群組 B：Atticus + Angel + 設計師（裝潢討論）
- 不同群組建立的行事曆，應邀請不同的人

### 設計方向（方案 A，已初步確認）
每個 LINE 群組在 Firestore 存一份獨立邀請名單（以 group_id 為 key）。

設定指令：
- `DoubleA 新增邀請人 Angel angelsu0923@gmail.com`
- `DoubleA 新增邀請人 王設計師 designer@gmail.com`
- `邀請名單` → 查看目前設定

### 尚未解決的問題

| 問題 | 嚴重度 | 說明 |
|------|--------|------|
| 待辦清單跨群組混在一起 | 🔴 高 | 不同群組的待辦都丟同一個 Google Tasks，早安摘要會全部混著顯示 |
| 早安/傍晚摘要只發一個群組 | 🔴 高 | 目前只存一個 chat_id，多群組後不確定發給誰 |
| 修改行事曆跨群組混淆 | 🟡 中 | 「最近一筆事件」是全域的，不分群組，可能改錯 |
| 確認訊息寫死「Angel 已收到邀請」 | 🟡 中 | 需改為動態顯示邀請人名稱 |
| 邀請名單管理權限 | 🟢 低 | 群組內任何人都能改設定，是否需要限制？ |

### 待決定
- 每個群組要不要有獨立的 Google Tasks 清單？
- 早安/傍晚摘要要對每個群組各自發嗎？

## 議題二：多人複製部署（產品化方向）

### 情境
哥哥看到 DoubleA 想要一個自己的，如何快速複製？

### 複製的瓶頸

| 項目 | 能否自動化 |
|------|-----------|
| LINE Developer 帳號 + channel | ❌ 必須手動（約 10 分鐘） |
| Google OAuth 授權 | ❌ 必須本人點擊授權（約 5 分鐘） |
| GCP 專案建立與帳單設定 | ⚠️ 可腳本化但仍需信用卡 |
| Cloud Run 部署 + Firestore + Scheduler | ✅ 可完全自動化 |

### 兩條路線

#### 路線 A：各自獨立部署（Independent Instance）
- 幫他人寫一份一鍵部署腳本 + 詳細教學
- 他自己申請 LINE + Google OAuth，其餘由腳本處理
- 優點：各自獨立，費用各付（每月 ~$0.5 USD）
- 缺點：對方需要一定技術門檻（GCP 帳號、信用卡）

#### 路線 B：多租戶共用後端（Multi-tenant SaaS）
- 一個 Cloud Run 服務多個家庭
- 每個家庭有自己的 LINE bot，共用你的後端
- Firestore 資料以 tenant_id 隔離
- 優點：對方只需設定 LINE + Google OAuth，其他你處理
- 缺點：你負責維護 + GCP 費用，架構複雜度提升

### 比較

| | 路線 A | 路線 B |
|--|--------|--------|
| 技術門檻（使用者） | 中 | 低 |
| 維護負擔（你） | 低 | 中 |
| 未來擴展性 | 差 | 好 |
| 適合情境 | 幫少數幾個人 | 想做成小產品 |

### 待決定
- 這個 bot 是否只留家用，或考慮推廣？
- 如果推廣，對象是技術人員還是一般使用者？
- 路線 A 還是路線 B？
